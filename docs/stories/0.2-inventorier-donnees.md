# Story 0.2 — Inventorier toutes les données

Statut : `user-validation`

## Objectif

Produire un état reproductible de Supabase, Auth, R2 et des données locales avant toute migration.

## Contexte

Le schéma versionné actuel contient `profiles`, `workspaces`, `workspace_members`, `workspace_invites`, `songs`, `setlists`, `setlist_songs` et `song_assets`. Dexie utilise la base globale `faderzero-pwa` en version 8 et attribue les anciennes lignes à `default-workspace` lors de la migration v7.

## Critères d’acceptation

- [x] Le SQL d’inventaire est enfermé dans une transaction `READ ONLY`.
- [x] Les comptages, tombstones, rôles, invitations et incohérences multi-espace sont couverts.
- [x] Les références audio nécessaires au rapprochement R2 sont exportables.
- [x] Deux exécutions sur la source figée produisent la même empreinte.
- [x] Le manifeste R2 complet (clé, taille, ETag, métadonnées) est exporté et rapproché.
- [x] Le schéma IndexedDB historique et le traitement de `default-workspace` sont documentés ; les comptages par appareil seront collectés avant Story 4.1.

## Tests

Revue statique du fichier `supabase/audit/epic-0-inventory.sql`, puis deux exécutions avec un compte SELECT-only. Comparer les exports SHA-256.

## Rollback

Aucun : le script est en lecture seule. En cas d’erreur, la transaction est abandonnée sans écriture.
