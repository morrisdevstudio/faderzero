# Story 1.5 — Garantir l’intégrité multi-espace

Statut : `user-validation`

## Objectif

Empêcher les relations entre espaces et rendre `workspace_id` immuable sans supprimer les éventuelles incohérences historiques.

## Critères d’acceptation

- [x] Les contraintes composites sont d’abord ajoutées `NOT VALID`.
- [x] Les anomalies sont journalisées en quarantaine sans suppression.
- [x] Les rattachements prouvables sont réparés et journalisés.
- [x] La validation est effectuée dans une migration séparée.
- [x] `workspace_id` devient immuable après validation.

## Données protégées

`songs`, `setlists`, `setlist_songs`, `song_assets`, y compris les tombstones.

## Rollback

Le journal conserve l’ancien et le nouveau rattachement. Les anciennes clés étrangères simples restent en place pendant la période de compatibilité.

## Validation

- Revue technique : expansion, quarantaine vide, validation et immuabilité vérifiées sur local et restauration.
- Validation utilisateur : Pause 1.
