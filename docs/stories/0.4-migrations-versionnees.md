# Story 0.4 — Installer les migrations versionnées

Statut : `user-validation`

## Objectif

Aligner le dépôt avec les dix migrations déjà enregistrées sur Supabase et prouver qu’elles reconstruisent une base neuve sans utiliser le script destructif `00_reset_faderzero.sql`.

## Contexte

La production contient dix migrations entre `20260717142736_bootstrap_schema` et `20260717142834_add_setlist_display_modes`. Le dépôt conserve leurs sources historiques dans `supabase/sql/`, mais pas encore dans `supabase/migrations/`.

## Critères d’acceptation

- [x] Les dix versions distantes ont un fichier local correspondant.
- [x] Aucun fichier de migration versionnée ne contient le reset de production.
- [x] La liste locale et la liste distante concordent, baseline comprise.
- [x] Une base PostgreSQL/Supabase neuve peut appliquer la chaîne complète.
- [x] Un rollback de test est documenté et ne cible que l’environnement isolé.
- [x] Typecheck, lint, 59 tests et build réussissent.

## Rollback

Les fichiers locaux peuvent être retirés avant commit. Toute réinitialisation est limitée au conteneur local explicitement identifié ; aucune commande `db reset` ou réparation d’historique ne sera lancée sur la production.

## Validation

- Revue technique : migrations, diff, lint, rollback/réapplication, deux restaurations et suite applicative validés.
- Validation utilisateur : requise à la Pause 0.
