# Story 4.3 — Partitionner les caches audio

Statut : `user-validation`

## Objectif

Empêcher un second compte de lire les audios hors ligne du précédent, sans perdre le cache historique.

## Critères d’acceptation

- [x] Cache nommé par utilisateur et workspace.
- [x] Les téléchargements, lectures, suppressions et vidages utilisent uniquement la partition active.
- [x] Le cache historique est copié sans être supprimé.
- [x] Chaque blob copié est vérifié contre `songAssets.sizeBytes`, puis relu après copie.
- [x] Un blob incohérent reste uniquement dans le cache historique.
- [x] L'app locale dispose d'une URL d'API audio et d'un Worker R2 local reproductible.
- [ ] Lecture audio réelle en ligne puis hors ligne validée sur l'appareil.

## Tests

- `npm test -- --run src/features/audio/audioCacheStore.test.ts` : 2/2.
- Copie correcte : blob présent dans une seule partition et toujours présent dans le cache historique.
- Taille incorrecte : copie refusée, original conservé.
- `npm run dev:audio` : Worker disponible sur le réseau local avec stockage R2 simulé et persistant.
- `GET http://192.168.1.23:8787/health` depuis l'origine de l'app : HTTP 200 et CORS autorisé.
- Préflight `PUT` depuis l'origine de l'app : HTTP 204 ; upload sans session : HTTP 401 attendu.

### Validation navigateur du 22/07/2026

- Compte de test local confirmé, chanson créée, MP3 synthétique de 17 180 octets importé puis lu avec succès (durée affichée : 00:01).
- Limite constatée : `http://192.168.1.23:5173` n'expose pas l'API Cache Storage (`cacheApiAvailable: false`). Le bouton « Mettre en cache hors ligne » ne peut donc pas être validé depuis cette origine HTTP non sécurisée.

## Rollback

`faderzero-audio-cache` est conservé pendant la fenêtre de compatibilité. Les nouvelles partitions peuvent être ignorées sans perte de l’ancien cache.
