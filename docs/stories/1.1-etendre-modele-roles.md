# Story 1.1 — Étendre le modèle de rôles

Statut : `user-validation`

## Objectif

Introduire `admin`, `member` et `guest`, convertir les cinq appartenances historiques `owner` en `admin` sans perte et conserver `owner` comme valeur compatible pendant deux versions et trente jours minimum.

## Contexte

- [Plan global](../PLAN_IMPLEMENTATION_GLOBAL_FADERZERO.md)
- [Audit de sécurité](../AUDIT_SECURITE_FADERZERO.md)
- [Inventaire Epic 0](../reports/EPIC_0_INVENTORY_2026-07-20.md)

## Critères d’acceptation

- [x] La contrainte accepte `owner`, `admin`, `member` et `guest`.
- [x] Chaque conversion `owner → admin` est journalisée avant mise à jour.
- [x] Le nombre total d’appartenances reste égal à 8 sur la copie de production.
- [x] Les nouveaux groupes utilisent `admin`.

## Données protégées

`public.workspace_members` : identifiants, rattachements, rôles et dates. Aucune ligne ne peut être supprimée.

## Tests

- Migration sur base locale puis sur la restauration de production.
- Comparaison des comptages et distribution des rôles avant/après.
- Rollback logique documenté par le journal de migration.

## Rollback

Le journal permet de rétablir précisément `admin → owner` pour les seules lignes migrées. La contrainte compatible conserve les deux valeurs pendant toute la période transitoire.

## Validation

- Revue technique : validée sur Supabase local et `faderzero_restore` ; test applicatif 12/12.
- Validation utilisateur : Pause 1.
