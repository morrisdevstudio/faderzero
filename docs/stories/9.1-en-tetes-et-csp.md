# Story 9.1 — En-têtes et CSP

Statut : `done`

## Objectif

Appliquer une politique de sécurité de contenu stricte à la PWA et garantir les en-têtes navigateur requis sans annoncer HSTS sur le listener HTTP interne.

## Contexte

- [Plan global](../PLAN_IMPLEMENTATION_GLOBAL_FADERZERO.md), Epic 9.
- [Audit de sécurité](../AUDIT_SECURITE_FADERZERO.md).
- Configuration de diffusion : `deploy/Caddyfile`.

Tailscale Serve termine TLS sur l’URL de tailnet, puis relaie vers Caddy en `https+insecure://localhost:8443`. Caddy sert le même site pour `localhost` et le nom Tailscale, avec un certificat interne ; HSTS est donc ajouté à une réponse HTTPS.

## Critères d’acceptation

- [x] La CSP bloque les scripts et styles inline ainsi que `unsafe-eval`.
- [x] `frame-ancestors 'none'`, `nosniff`, `no-referrer` et une Permissions Policy restrictive sont configurés.
- [x] Un contrôle automatisé bloque une régression de ces en-têtes dans la CI.
- [x] Tailscale Serve force HTTPS et transmet HSTS uniquement sur les réponses HTTPS ; une preuve de recette est archivée.

## Données protégées

Aucune table, objet R2, base IndexedDB, cache ou file de synchronisation n’est modifiée.

## Tâches

## Tests

### Automatiques

- Commande : `npm run security:headers`
- Résultat attendu : la CSP et tous les en-têtes requis sont validés ; aucun `unsafe-eval`, `unsafe-inline` ou HSTS sur le listener HTTP n’est accepté.

### Manuels

- Parcours : appeler l’URL Tailscale en HTTPS, puis inspecter les en-têtes de la réponse.
- Résultat attendu : HTTPS expose HSTS, CSP, `X-Content-Type-Options`, `Referrer-Policy` et `Permissions-Policy`.
